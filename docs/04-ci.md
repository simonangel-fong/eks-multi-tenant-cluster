---

- **Goal:** on every push to `master`, run tests, build the API image, and push an immutable, SHA-tagged image to Docker Hub — so ArgoCD (phase 7) can pin to a specific tag and roll forward deterministically.
- **Done when:**
  - `git push origin master` triggers a green workflow.
  - Tests run against a real Postgres (matching how pytest runs locally).
  - Image lands in Docker Hub tagged with the commit SHA (no `:latest` on `master`).
  - The SHA in the registry equals `git rev-parse HEAD`.

---

## layout

```
.github/
└─ workflows/
   └─ app-ci.yml     # test + build + push on app/** changes
```

**One pipeline, one job of work.** No `helm-ci.yml` yet — `helm lint` runs locally as part of the phase-4 dev loop and there's no chart drift to catch in CI. No `terraform-ci.yml` — there is no `terraform/` directory yet. Add those workflows when the code they'd lint actually exists.

## workflow: `app-ci.yml`

Triggered on:

- `push` to `master`, paths: `app/**`, `.github/workflows/app-ci.yml`
- `pull_request` targeting `master`, same paths (test + build only, no push)

Jobs:

| Job     | Purpose                                                                                          | Runs on PRs? | Pushes image? |
| ------- | ------------------------------------------------------------------------------------------------ | ------------ | ------------- |
| `test`  | `uv sync`, boot Postgres via `services:`, run Flyway migrate, `uv run pytest -v`                 | yes          | no            |
| `build` | multi-stage Docker build (same `app/Dockerfile`), Buildx cache, tag with `${sha}`                | yes          | no on PR      |
| `push`  | needs `build`; login to Docker Hub; push `${sha}` + `${short-sha}`; **only on `push` to `master`** | no         | yes           |

**Postgres for `test`.** GitHub Actions `services:` runs `postgres:16-alpine` on the runner. Flyway runs as a one-shot step against it before pytest — same order as `docker-compose.yml`, so local and CI stay identical.

```yaml
services:
  postgres:
    image: postgres:16-alpine
    env: { POSTGRES_USER: voting, POSTGRES_PASSWORD: voting, POSTGRES_DB: voting }
    ports: ["5432:5432"]
    options: >-
      --health-cmd "pg_isready -U voting -d voting"
      --health-interval 3s --health-timeout 3s --health-retries 10
```

pytest also needs `voting_test` (created locally by `sql/initdb/00_create_test_db.sql`). CI mirrors that with a single `psql -c 'CREATE DATABASE voting_test'` step before Flyway.

**Build caching.** Use `docker/build-push-action@v6` with `cache-from: type=gha` / `cache-to: type=gha,mode=max`. The `uv sync` layer is the expensive one — caching drops warm builds significantly.

## image tagging

**Rule: `master` produces exactly one immutable tag per commit, and nothing else.**

| Tag                 | When                    | Mutable? | Used by                             |
| ------------------- | ----------------------- | -------- | ----------------------------------- |
| `${sha}` (full 40)  | every `master` build    | no       | ArgoCD (`values-prod.yaml`, phase 7)|
| `${short-sha}` (7)  | every `master` build    | no       | humans reading `kubectl describe`   |
| ~~`latest`~~        | **never on `master`**   | —        | —                                   |

`:latest` breaks ArgoCD sync detection — the tag doesn't change, so ArgoCD can't tell a rollout is needed. CI's job is to keep the registry filled with valid SHAs, not to auto-bump the values file (that's phase 7's call — bot commit or Argo Image Updater).

**Docker Hub for now.** Registry is `simonangelfong/voting-api`. When EKS lands (phase 6), we migrate to ECR — but that's a phase-6 change, not a CI decision. Today's target is the same registry the chart already points at.

## secrets & permissions

- `GITHUB_TOKEN` — default, read-only for content.
- Repo secrets: `DOCKERHUB_USERNAME`, `DOCKERHUB_TOKEN` — used only by the `push` job.
- Job `permissions:` block scoped narrowly:
  ```yaml
  permissions:
    contents: read
  ```
- Never `permissions: write-all`. Never a personal access token.

## delivery phase

| #   | phase             | description                                                                                  |
| --- | ----------------- | -------------------------------------------------------------------------------------------- |
| 5.1 | scaffold          | `.github/workflows/app-ci.yml` — `test` job only; PR + `master`; green against current codebase |
| 5.2 | build job         | add `build` job with Buildx + gha cache; runs on PRs but doesn't push                        |
| 5.3 | push to Docker Hub | add `push` job gated on `push` to `master`; verify a new SHA tag appears after merge         |

**Commit at every sub-phase.** Never merge a broken workflow to `master` — use a feature branch and let the PR run of the workflow itself prove it works before merging.

## Development

```sh
# lint the workflow file locally before pushing
docker run --rm -v ${PWD}:/repo --workdir /repo rhysd/actionlint:latest -color


# render the image the same way CI will
docker buildx build --load -t voting-api:ci-check app/

# run pytest against a fresh compose stack — mirrors the CI `test` job
docker compose down -v
docker compose up -d postgres
docker compose up flyway flyway-test
cd app && uv sync && uv run pytest -v
```

**After first green push to `master`:**

```sh
git rev-parse HEAD
# 3f2a1e8...

docker pull simonangelfong/voting-api:3f2a1e8...
docker inspect simonangelfong/voting-api:3f2a1e8... --format '{{.Created}}'
```

The SHA in the registry must equal `git rev-parse HEAD` on `master`. If it doesn't, either the workflow didn't run or it pushed the wrong tag — treat it as a hard failure of the phase.
